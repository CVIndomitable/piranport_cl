## P0 — 必须立即修复（5个）

### 1. 区块强制加载泄漏 — 服务器运行越久越卡
**文件**: `AircraftEntity.java`
**问题**: 侦察机使用 `ServerLevel.setChunkForced()` 加载 3x3 区块，但：
- 没有重写 `Entity.remove(RemovalReason)` 作为清理安全网
- `/kill` 命令、服务端崩溃、实体意外移除等路径都会跳过 `releaseAllForcedChunks()`
- `setChunkForced` 是持久化到存档的操作，泄漏后重启也不会恢复
- `lastForcedChunkX/Z` 是 runtime 字段（不存 NBT），重载后无法释放旧区块

**影响**: 每次泄漏永久加载9个区块，长期运行后服务器内存膨胀、TPS持续下降

**建议修复**:
```java
@Override
public void remove(RemovalReason reason) {
    if (!level().isClientSide()) {
        releaseAllForcedChunks();
        if (ownerUUID != null) {
            ReconManager.endRecon(ownerUUID);
        }
    }
    super.remove(reason);
}
```
同时将 `lastForcedChunkX/Z` 写入 NBT，并考虑使用 NeoForge 的 `ForgeChunkManager` 替代原始 `setChunkForced`。

### 2. FlammableEffect 单例共享状态 — 多人服务器爆炸行为混乱
**文件**: `FlammableEffect.java:22`
**问题**: `explosionTimer` 是实例变量，但 `MobEffect` 在注册系统中是**单例**。所有带此效果的玩家共享同一个计时器，3个玩家同时带效果时爆炸频率提高3倍。

**建议修复**: 移除实例变量 `explosionTimer`，改用 `entity.tickCount % 39 == 0` 替代自增计时器，或通过 `PersistentData` 存储每个实体的计时器状态。

### 3. ReconControlPayload 输入未校验 — 恶意客户端可瞬移飞机
**文件**: `ReconControlPayload.java:34-39`
**问题**: 客户端发来的 `dx/dy/dz` 未做范围校验。恶意客户端可发送 `Float.MAX_VALUE` 或 `NaN`，导致侦察机瞬移到极远位置触发区块加载，或 NaN 导致区块损坏。

**建议修复**:
```java
float dx = Mth.clamp(payload.dx(), -1.0f, 1.0f);
float dy = Mth.clamp(payload.dy(), -1.0f, 1.0f);
float dz = Mth.clamp(payload.dz(), -1.0f, 1.0f);
if (Float.isNaN(dx) || Float.isNaN(dy) || Float.isNaN(dz)) return;
```

### 4. FlightGroupData 反序列化无大小上限 — OOM 崩溃攻击
**文件**: `FlightGroupData.java:95-112`
**问题**: StreamCodec 解码时直接用网络读取的 `size` 作为循环次数，无上限校验。恶意客户端发送 `size=Integer.MAX_VALUE` 可触发 `OutOfMemoryError`。

**建议修复**:
```java
int size = ByteBufCodecs.VAR_INT.decode(buf);
if (size < 0 || size > MAX_GROUPS) throw new DecoderException("Too many groups: " + size);

int cnt = ByteBufCodecs.VAR_INT.decode(buf);
if (cnt < 0 || cnt > 16) throw new DecoderException("Too many slot indices: " + cnt);
```

### 5. 侦察机减速效果可能永久锁定玩家
**文件**: `AircraftEntity.java:258-259`
**问题**: 对玩家施加 `MOVEMENT_SLOWDOWN` amplifier 9，duration `Integer.MAX_VALUE`。如果服务端崩溃、实体被 `/kill`，`removeEffect` 不会被调用，玩家永久无法移动。

**建议修复**:
1. 在 `remove(RemovalReason)` 中对 owner 移除减速效果（与 P0 #1 合并）
2. 在 `GameEvents.onPlayerTick` 中增加安全检查：如果玩家有 9 级减速但没有活跃的侦察机，自动移除
3. 考虑使用自定义 MobEffect 替代原版 `MOVEMENT_SLOWDOWN`

### 7. FlightGroupUpdatePayload 内容未验证
**文件**: `FlightGroupUpdatePayload.java:34-46`
**问题**: 客户端发来的 `FlightGroupData` 仅验证了槽位范围和物品类型，未验证 `slotIndices` 是否在武器槽范围内。恶意客户端注入 index=999 可导致后续 `IndexOutOfBoundsException`。

**建议修复**: 在 `handle()` 中验证 `FlightGroupData` 的每个 `slotIndices` 元素都在 `0 <= index < shipType.weaponSlots` 范围内，限制 `groups.size() <= MAX_GROUPS`，白名单校验 `payloadType` 字符串。

### 8. AttackMode StreamCodec 无边界检查
**文件**: `FlightGroupData.java:39-40`
**问题**: `AttackMode.values()[i]` 无边界保护，恶意 ordinal 值触发 `ArrayIndexOutOfBoundsException`。对比 `FireAction.STREAM_CODEC` 已正确做了防御。

**建议修复**:
```java
ByteBufCodecs.VAR_INT.map(i -> {
    var vals = values();
    return i >= 0 && i < vals.length ? vals[i] : FOCUS;
}, Enum::ordinal);
```

### 9. AP弹护甲修改器异常时不被清理

```java
try {

```
