# v0.0.7-alpha — Automation & Aviation+（自动化兼容 + 航空进阶）

**目标：三种加工站的漏斗/管道/AE2 自动化兼容；航空系统进阶（侦察机、空战、编队跟随）。**

**设计原则：自动化兼容不改变玩家手动操作体验，仅扩展机器接口；航空进阶在 v0.0.4 四重防御框架内扩展，不引入新的实体管理风险。**

### 1.2 石磨 — 测试确认

石磨已实现漏斗兼容，本版本仅做 AE2 兼容性测试确认：

**测试项：**
- AE2 Pattern Provider 面朝石磨输入面，推入原料后是否瞬时产出

### 1.3 厨锅 — 输入输出面 IItemHandler

**改动范围：**

1. **CookingPotBlockEntity 注册 IItemHandler capability**
   - 输入面（顶部 + 侧面）：暴露 9 个输入槽，外部可推入食材
   - 输出面（底部）：暴露 1 个输出槽，外部可抽取产物
   - 方向判定参考原版熔炉逻辑：`Direction.DOWN` → 输出，其余 → 输入

2. **注册 capability**
   ```java
   // 在 RegisterCapabilitiesEvent 中注册
   event.registerBlockEntity(
       Capabilities.ItemHandler.BLOCK,
       ModBlockEntityTypes.COOKING_POT.get(),
       (be, direction) -> direction == Direction.DOWN
           ? be.getOutputHandler()
           : be.getInputHandler()
   );
   ```

3. **输入限制**
   - `InputHandler.insertItem()` 检查：槽位已有物品时只接受相同物品堆叠
   - 不限制物品种类（任何物品都可推入，配方匹配由 tick 逻辑处理）

4. **输出行为**
   - 烹饪完成后产物进入输出槽（现有逻辑不变）
   - 漏斗/管道从底部抽取输出槽
   - 不做自动弹出（保留手动取出作为无漏斗时的操作方式）

5. **热源检测不变** — 自动化只解决物品进出，热源仍需玩家手动搭建

**数据流：**
```
漏斗/管道 → [顶部/侧面] → 9个输入槽 → tick匹配配方 → 计时 → 产物进输出槽 → [底部] → 漏斗/管道抽取
```

### 1.4 砧板 — 农夫乐事式自动化

**设计参考：** Farmer's Delight 的 Cutting Board 自动化方案。

**改动范围：**

1. **CuttingBoardBlockEntity 注册单槽 IItemHandler capability**
   - 暴露 1 个槽位：当前放置的待切割物品
   - `insertItem()`：砧板为空时接受物品（等同于手持右键放入）
   - `extractItem()`：返回空（产物通过弹出到世界的方式输出，不走 capability）
   - 所有方向均可输入（漏斗从任意面推入）

2. **发射器切割支持**
   - 监听 `BlockEvent` 或在 `CuttingBoardBlock.neighborChanged()` 中检测
   - 当相邻发射器激活时（`DispenserBlock` 发出红石信号），对砧板执行一次切割
   - 实现方式：监听 `NeoForge.EVENT_BUS` 的 `BlockEvent.NeighborNotifyEvent`，或直接在砧板的 `neighborChanged` 中检测
   - 更稳妥的方案：**自定义发射器行为**（`DispenserBlock.registerBehavior`），当发射器内有"刀"物品时对前方砧板执行切割

3. **切割触发逻辑**
   ```java
   // 被发射器激活时
   public boolean performCut() {
       if (currentItem.isEmpty()) return false;
       CuttingBoardRecipe recipe = findRecipe();
       if (recipe == null) return false;
       cuttingProgress++;
       if (cuttingProgress >= recipe.getCuts()) {
           // 弹出产物到世界（现有逻辑）
           popResults(recipe);
           currentItem = ItemStack.EMPTY;
           cuttingProgress = 0;
       }
       setChanged();
       level.sendBlockUpdated(...);
       return true;
   }
   ```

4. **产物收集** — 产物弹出到世界后，由砧板下方的漏斗收集（与手动操作一致）

5. **手动操作不变** — 玩家右键放入/切割/Shift取回的逻辑完全保留

**自动化数据流：**
```
漏斗 → [任意面] → 砧板单槽 → 发射器激活 → 切割进度+1 → 达标后产物弹出到世界 → 下方漏斗收集
```

**关于"刀"的设计决策：**

| 方案 | 描述 | 优劣 |
|------|------|------|
| A. 任意发射器激活 | 相邻发射器收到红石信号就切一刀 | 简单，但不太合理（空发射器也能切？） |
| B. 发射器内需要"刀"物品 | 注册自定义 DispenseBehavior，发射器内有铁剑/铁锭时触发 | 合理，但需要新注册 |
| C. 发射器内需要专用切刀物品 | 新增"菜刀"物品，注册 DispenseBehavior | 最合理，但增加物品 |

**建议采用方案 A**（任意发射器激活即切割）。理由：农夫乐事也是这个方案，玩家已有认知；省去额外物品注册；方案 B/C 可以作为后续打磨放到更晚版本。

## 二、Aviation+（航空进阶）

### 2.1 侦察机

**设计目标：** 放出侦察机后切换到侦察机视角，实现远距离目视侦察。

**核心机制：**

| 要素 | 设计 |
|------|------|
| 飞机类型 | 新增 `RECON`（侦察机），新物品 `recon_squadron` |
| 放飞方式 | 与其他飞机一致（编组GUI配置 → 右键放飞） |
| 视角切换 | 放飞后自动切换玩家视角到侦察机实体（类似骑乘幻翼的第三人称视角） |
| 操控 | 玩家 WASD 控制侦察机飞行方向，空格上升，Shift下降 |
| 退出 | 按 V 键或右键退出侦察视角，回到玩家本体 |
| 区块加载 | 侦察机周围 3×3 区块临时加载（`ForgeChunkManager.forceChunk`） |
| 限制 | 同一时间只能操控一架侦察机；操控期间玩家本体无法移动/攻击 |

**状态机扩展：**
```
LAUNCHING → RECON_ACTIVE（玩家操控中）→ RETURNING → REMOVED
```
- `RECON_ACTIVE` 是新增状态，仅侦察机使用
- 侦察机不进入 CRUISING/ATTACKING，没有攻击能力
- 四重防御机制照常生效（滞空上限、距离上限等）

**AircraftInfo 扩展：**

| 字段 | 侦察机值 |
|------|---------|
| aircraftType | RECON |
| fuelCapacity | 1500（比攻击机多，滞空更久） |
| ammoCapacity | 0（无弹药） |
| panelDamage | 0（无攻击） |
| panelSpeed | 2.0（最快，纯侦察用途） |
| weight | 8（最轻） |

**弹药需求：** 仅航空燃料，不需要其他弹药。编组GUI中弹种自动设为燃料，无需手动配置。

**实现关键点：**
- 视角切换：`Minecraft.getInstance().setCameraEntity(reconEntity)` （客户端）
- 操控同步：客户端每 tick 读取输入 → C2S Payload 发送方向 → 服务端更新实体运动
- 退出时：`setCameraEntity(player)` 恢复 + 侦察机自动返航
- 区块加载：服务端在侦察机 tick 中调用 `ForgeChunkManager`，卸载在 RETURNING 时释放

**网络包：**
- `ReconControlPayload`（C2S）：玩家输入方向（dx, dy, dz 归一化向量）
- `ReconStartPayload`（S2C）：通知客户端切换视角到指定实体 ID
- `ReconEndPayload`（S2C）：通知客户端恢复视角

### 2.2 空战

**设计目标：** 战斗机可以攻击敌方飞机实体，形成制空权争夺。

**机制：**

| 要素 | 设计 |
|------|------|
| 攻击目标扩展 | 战斗机 (FIGHTER) 的 `resolveTarget()` 增加对 `AircraftEntity` 的索敌 |
| 优先级 | 锁定目标 > 敌方飞机 > 敌对生物（敌方飞机优先于地面目标） |
| "敌方"判定 | `AircraftEntity.getOwnerUUID() != 自身.getOwnerUUID()`（不同玩家的飞机） |
| 攻击行为 | 复用 FIGHTER 现有近战逻辑：追尾接近→距离<2.5格时每20tick咬伤 |
| 飞机血量 | AircraftEntity 新增 `health` 字段，默认 20HP |
| 击落 | 血量归零时强制 REMOVED + 物品**不返还**（击落 = 损失） |
| 友军保护 | 同一玩家的飞机之间不互相攻击 |

**数值：**

| 飞机 | 血量 | 被战斗机击杀耗时（理论） |
|------|------|----------------------|
| 战斗机 | 20 | ~9秒（panelDmg 18, 每20tick一次） |
| 俯冲轰炸机 | 15 | ~7秒 |
| 鱼雷轰炸机 | 15 | ~7秒 |
| 水平轰炸机 | 12 | ~5秒 |
| 侦察机 | 10 | ~4秒（最脆弱） |

> 血量设计原则：战斗机最耐打，轰炸机中等，侦察机最脆弱。符合舰R中制空权的战术定位。

**AircraftEntity 改动：**
- 新增 `health` 运行时字段（不持久化，每次放飞重置为满血）
- 覆写 `hurt(DamageSource, float)` 扣血
- `health <= 0` 时进入 `REMOVED` 状态，**不调用 `returnItemToOwner()`**（物品损失）
- 击落时播放爆炸粒子 + 音效

### 2.3 编队跟随侦察机

**设计目标：** 攻击编组可以跟随侦察机移动，实现远距离投送。

**机制：**

| 要素 | 设计 |
|------|------|
| 跟随触发 | 编组GUI新增"跟随"模式（与集火/分散并列第三种攻击模式） |
| 跟随行为 | CRUISING 状态时，飞机环绕侦察机盘旋（而非环绕玩家） |
| 切换攻击 | 玩家对跟随编组按 P 锁定目标后，编组从 CRUISING → ATTACKING |
| 侦察机返航时 | 跟随编组自动切为环绕玩家盘旋（退化为普通 CRUISING） |
| 前提 | 必须有一架侦察机在 RECON_ACTIVE 状态 |

**FlightGroupData 扩展：**
```java
attackMode: FOCUS | SPREAD | FOLLOW  // 新增 FOLLOW
```

**AircraftEntity 改动：**
- `tickCruising()` 中检查 `attackMode == FOLLOW`
- 是 → 找到同一玩家的 RECON_ACTIVE 侦察机实体，以其为中心盘旋
- 否 → 维持现有以玩家为中心的盘旋逻辑
- 侦察机不存在或已返航 → 退化为以玩家为中心
