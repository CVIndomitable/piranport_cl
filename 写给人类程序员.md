## 三、核心架构模式

### 3.1 注册流程（必须理解）

所有注册对象都走 `DeferredRegister`，在 `PiranPort.java` 构造函数中统一注册到 `modEventBus`。**顺序很重要**：DataComponents 必须先于 Items 注册，因为 Items 里会引用 Component。

```
DataComponents → Blocks → Items → CreativeTabs → MenuTypes
→ EntityTypes → MobEffects → BlockEntityTypes → RecipeTypes
```

### 3.2 功能方块的标准四件套

每个带 GUI 的功能方块需要同时实现：

| 类 | 职责 |
|----|------|
| `XxxBlock.java` | 方块逻辑、FACING 属性、右键开GUI、移除时掉落物品 |
| `XxxBlockEntity.java` | 存储数据、实现 MenuProvider |
| `XxxMenu.java` | 格子布局、shift点击逻辑 |
| `XxxScreen.java` | 渲染 GUI 背景和进度条 |

还需要在注册类中注册：
- `ModBlockEntityTypes.java`：`BlockEntityType`
- `ModMenuTypes.java`：`MenuType`（用 `IMenuTypeExtension.create`）
- `ClientEvents.java`：`MenuScreens.register`

### 3.3 自定义配方系统

三种自定义配方类型（石磨/厨锅/砧板）共用同一套结构：

```
RecipeType（标识符）
RecipeSerializer（JSON读写 + 网络读写）
Recipe<XxxInput>（配方逻辑，implements Recipe）
XxxRecipeInput（配方匹配时传入的容器，implements RecipeInput）
```

配方 JSON 放在 `data/piranport/recipe/` 下，格式见下方示例。
配方类型和序列化器一起在 `ModRecipeTypes.java` 中注册。

### 3.4 食物放置系统

食物物品通过 `PlaceableInfo` DataComponent 声明"可放置为方块"：

```java
.component(ModDataComponents.PLACEABLE_INFO.get(), new PlaceableInfo("plate", 3))
// "plate"/"bowl"/"cake" 对应三种通用形状方块
// 3 = 总份数
```

`ModFoodItem.useOn()` 检测到 Shift+右键地面时，放置对应的 `PlaceableFoodBlock`（plate/bowl/cake 三选一），并初始化 `PlaceableFoodBlockEntity` 存储食物ID和份数。右键食用时调用 `BlockEntity.eat()` 消耗份数。

`PlaceableFoodBlock` 不能用单个父类注册（NeoForge codec 限制），必须用三个静态内部类各自实现 `simpleCodec`。

### 3.5 酿造配方

**不走 JSON**，在代码中注册。监听 `RegisterBrewingRecipesEvent`（注意：这是 **NeoForge 游戏总线**事件，不是 mod 总线事件），调用 `event.getBuilder().addRecipe()`。

## 四、已踩坑清单（必读）

| 坑 | 错误写法 | 正确写法 |
|----|---------|---------|
| 酿造配方注册 | `modEventBus.addListener(RegisterBrewingRecipesEvent)` | `NeoForge.EVENT_BUS.addListener(...)` |
| 酿造配方旧 API | `BrewingRecipeRegistry.addRecipe(recipe)` | `event.getBuilder().addRecipe(recipe)` |
| 食物饱和度 | `food.saturationModifier()` | `food.saturation()` |
| 熔炉热源检测 | `AbstractFurnaceBlockEntity.isLit()`（private）| `bs.hasProperty(BlockStateProperties.LIT) && bs.getValue(...LIT) && bs.getBlock() instanceof AbstractFurnaceBlock` |
| 方块掉落物品 | `Containers.dropContents(level, pos, be)`（需实现 Container）| 手动 loop `itemHandler.getStackInSlot(i)` + `Containers.dropItemStack()` |
| PlaceableFoodBlock codec | 单个基类用 `simpleCodec(Base::new)` | 三个静态内部类各自实现 `simpleCodec(Plate::new)` 等 |

## 五、如何添加新内容

### 添加一个普通物品

1. `ModItems.java`：`ITEMS.registerSimpleItem("id")`
2. `models/item/id.json`：通常复制一个已有的 `item/generated` JSON
3. `textures/item/id.png`：16×16 贴图
4. `zh_cn.json` + `en_us.json`：添加 `"item.piranport.id": "名字"`
5. `ModCreativeTabs.java`：加到 Creative Tab

### 添加一个食物（带放置功能）

1. 同上，但用 `ModFoodItem` 替代普通 `Item`，传入 `FoodProperties` 和 `PlaceableInfo`
2. 在 `ModCreativeTabs.java` 归类到食物分组
3. 写厨锅/砧板/石磨配方 JSON（见配方格式）

### 添加一个厨锅配方

在 `data/piranport/recipe/` 新建 JSON：

```json
{
  "type": "piranport:cooking_pot",
  "ingredients": [
    {"item": "piranport:flour"},
    {"item": "piranport:flour"},
    {"item": "piranport:butter"}
  ],
  "result": {"id": "piranport:toast_bread", "count": 1},
  "cookingTime": 200
}
```

- `cookingTime` 单位是 tick（20tick = 1秒），不填默认 200
- 输入最多 9 个，与厨锅格子数量一致

### 添加一个石磨配方

```json
{
  "type": "piranport:stone_mill",
  "ingredients": [{"item": "minecraft:wheat"}],
  "result": {"id": "piranport:flour", "count": 1}
}
```

### 添加一个砧板配方

```json
{
  "type": "piranport:cutting_board",
  "ingredient": {"item": "minecraft:bread"},
  "result": {"id": "piranport:toast_bread_slices", "count": 3},
  "cuts": 3
}
```

### 添加一个新作物

1. `ModBlocks.java`：注册 `FourStageCropBlock` 或 `ThreeStageCropBlock`，传入种子物品 Supplier
2. `ModItems.java`：注册种子（`ItemNameBlockItem`）和产物物品
3. `blockstates/crop_id.json`：4或3阶段 variant，引用 `block/crop/crop_id_stageN`
4. `models/block/crop/crop_id_stageN.json`（N=0,1,2,3）：用 `block/crop` 父模型
5. `textures/block/` 放贴图（每阶段一张）
6. `loot_table/blocks/crop_id.json`：成熟时掉落产物+种子，未成熟只掉种子
7. `zh_cn.json` 添加翻译

### 添加一个带 GUI 的功能方块

需要同时写：
1. `XxxBlock.java` → `ModBlocks.java` 注册
2. `XxxBlockEntity.java` → `ModBlockEntityTypes.java` 注册
3. `XxxMenu.java` + `XxxScreen.java` → `ModMenuTypes.java` 注册，`ClientEvents.java` 绑定
4. blockstate/model JSON，lang 翻译，loot table

## 六、事件总线区别

NeoForge 有两条总线，注册错了会崩溃：

| 总线 | 用于 | 使用方式 |
|------|------|---------|
| `modEventBus`（构造函数传入）| Mod 生命周期事件：注册、FML Setup、DataGen | `modEventBus.addListener(...)` |
| `NeoForge.EVENT_BUS` | 游戏运行时事件：玩家交互、方块破坏、酿造注册等 | `NeoForge.EVENT_BUS.addListener(...)` |

客户端事件类用注解标注总线：
```java
@Mod.EventBusSubscriber(modid = MOD_ID, bus = Bus.MOD, value = Dist.CLIENT)
```

## 七、构建与运行

```bash
./gradlew runClient    # 启动游戏客户端（开发环境）
./gradlew runData      # 运行 DataGen 生成 JSON
./gradlew build        # 构建 JAR → build/libs/piranport-0.0.3.jar
```

> 每次会话结束后，Stop Hook 会自动执行 `./gradlew build` 并复制 JAR 到 Minecraft mods 目录。
