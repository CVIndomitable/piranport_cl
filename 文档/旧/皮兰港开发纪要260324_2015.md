# 皮兰港模组 · 开发纪要

> 文档生成时间：2026-03-24
> 记录范围：从策划案评审到 v0.2.0 规划确定

## 1. 原始策划案评审

### 1.1 策划案概况

收到策划文档《总策划案-20220816-更新港区方块.docx》，这是一份 Minecraft 战舰少女R 模组的总策划案（重置版），由"皮兰港"团队编写于 2022 年 7 月，策划案主编 de Nestu，主催轻航独角受。

策划案覆盖了以下子系统：

| 系统 | 完成度 | 复杂度 |
|------|--------|--------|
| 舰装系统（核心、皮肤、负重、变身） | 较高 | 极高 |
| 战斗系统（炮击/雷击/航空/反潜/导弹） | 中等 | 极高 |
| 资源系统（矿物、植物） | 较高 | 低 |
| 方块系统（装饰、功能机器） | 框架 | 中等 |
| 食物-Buff 系统（烹饪、Buff） | 较高 | 中等 |
| NPC 系统（深海、舰娘） | 几乎空白 | 高 |
| 海图系统（副本、深海世界） | 框架 | 极高 |

### 1.2 可行性评估结论

**整体评价：设计雄心大，落地风险高。**

**主要风险：**

1. **范围过大** — 七八个大系统，每个有大量子模块，相当于中型商业游戏的内容量，但团队是同人社团编制。缺少"最小可玩版本"定义。

2. **技术难度被低估** — 侦察机视角切换（留假人模型+强加载区块）、鱼雷/导弹/战斗机 AI、舰装栏 GUI 深度注入原版背包、三层外观渲染系统（皮肤+船体+模块化武器），每一个都是独立的高难度课题。

3. **大量"待补充"** — NPC 系统、海图副本、深海世界维度、剧情/美术/音乐章节基本为空，策划实际完成度约 50-60%。

4. **数值体系未验证** — 负重上限 112/64/40 的设计缺乏典型配装验证案例，"火炮伤害数值设计的目的"标注为"待补充"。

**值得肯定的亮点：**

- 舰装栏和变身系统逻辑完整，锁定/解锁和弹药库惩罚机制考虑周到
- 烹饪系统（锅的热源判定、产物弹出优先级、碗架渲染）细节充分
- 资源系统克制，没有盲目堆砌新矿物
- 版本分阶段推进思路正确

## 2. 技术选型

### 2.1 原方案问题

策划案基于 **Minecraft 1.16.5 + Forge API**，原有工程代码已丢失。

### 2.2 版本选型调研

经调研 2026 年 Minecraft 模组开发生态现状，结论如下：

| 选项 | 状态 | 推荐度 |
|------|------|--------|
| 1.16.5 Forge | 已过时，生态衰退 | ❌ |
| 1.20.1 Forge | 遗留版本，仍有大量模组但不再活跃发展 | ⚠️ |
| 1.21.x NeoForge | 当前主流，16000+ 模组，生态活跃 | ✅ |
| Fabric | 轻量级，适合 Vanilla+ 而非重度内容模组 | ❌ |

**关键事实：**

- 2023 年 Forge 核心团队分裂，主维护者 cpw 等创建 NeoForge
- NeoForge 已成为 1.20.5+ 版本的事实标准，所有主流大型整合包（ATM10、FTB NeoTech 等）均已迁移
- 性能提升显著：启动时间 -32%、内存占用 -19%、TPS +10%
- Mojang 从 26.1 快照开始移除代码混淆，NeoForge 已提前适配

### 2.3 最终选型

```
平台：NeoForge 21.11.x
Minecraft 版本：1.21.11
Java：21
Gradle 插件：ModDevGradle
```

选择 1.21.11 而非 1.21.1 的原因：NeoForge 21.11 是截至决策时的最新稳定版本，API 更成熟，且已有 MDK 模板。

## 3. MVP 实施 (v0.1.0-alpha)

### 3.1 设计思路

从庞大的策划案中裁剪出**一条完整的游戏循环**作为最小可玩版本：

```
挖矿 → 冶炼 → 合成装备 → 装进舰装核心 → 变身 → 开炮打怪
```

### 3.2 MVP 范围（5 个 Phase）

| Phase | 内容 | 验证标准 |
|-------|------|---------|
| Phase 1 | 物品/方块注册、Creative Tab | Creative Tab 可见所有物品 |
| Phase 2 | 矾土矿 + 盐块世界生成 | 新世界能找到矿物 |
| Phase 3 | 合成表 + 熔炉配方 | 工作台/熔炉合成正常 |
| Phase 4 | 舰装核心 GUI（DataComponents + Container） | 右键打开 GUI，放入武器/弹药，显示负重 |
| Phase 5 | 火炮投射物 + 变身状态 | 变身后发射炮弹，HE 爆炸 / AP 穿甲 |

### 3.3 关键技术决策

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 物品数据存储 | DataComponents（非旧版 NBT） | 1.21.x 标准做法，Codec 序列化更规范 |
| 世界生成 | Data-Driven JSON + BiomeModifier | 不需要写 Java 注册 Feature |
| 网络同步 | CustomPacketPayload + RegisterPayloadHandlersEvent | NeoForge 21.x 新 API |
| 变身系统 | 简化为 Shift+右键切换 | 原策划的动画/无敌帧/buff清除留到后续 |

### 3.4 MVP 中明确排除的内容

鱼雷、航空、导弹、反潜、皮肤/模型渲染、NPC、海图副本、深海世界、食物/Buff、装饰方块、功能方块、树木作物、剧情系统。

### 3.5 完成状态

**✅ v0.1.0-alpha 已完成。**

## 4. v0.2.0 规划 (Torpedo)

### 4.1 优先级排序

完成 MVP 后，通过评估确定了后续系统的优先级：

| 优先级 | 系统 | 理由 |
|--------|------|------|
| 1 | 鱼雷系统 | 第二武器体系，丰富战斗玩法 |
| 2 | 舰装栏 GUI 完善 + 负重调优 | 核心交互需打磨 |
| 3 | 食物烹饪 + Buff | 后勤体系 |
| 4 | 矿物/资源扩充 + 装饰方块 | 内容填充 |
| 5 | 皮肤/模型渲染 | 美术向，技术复杂，优先级最低 |

### 4.2 v0.2.0 范围（Phase 6-10）

#### Phase 6: 鱼雷物品与实体注册

新增 5 个物品 + 1 个实体：

- 533mm 标准鱼雷 / 610mm 氧气鱼雷（弹药，可堆叠 16）
- 双联装 / 三联装 / 四联装鱼雷发射器
- TorpedoEntity 投射物

#### Phase 7: 鱼雷物理与 AI

鱼雷不是抛物线炮弹，而是**水面航行的自驱动实体**。核心 tick 逻辑：

```
空气 + 下方是水 → 垂直速度清零，水平航行（贴水面）
在水中 → 向上加速（上浮）
其他 → 向下加速（下坠）
```

命中附加"进水"debuff（自定义 MobEffect，每秒 1 点魔法伤害）。

数值设定：

| 弹药 | 伤害 | 航速 | 航程 |
|------|------|------|------|
| 533mm | 18 | 1.2 b/t | 10s |
| 610mm | 28 | 1.0 b/t | 15s |

#### Phase 8: 鱼雷发射器机制

发射器按使用键扇形发射多枚鱼雷：

| 连装数 | 散布角度 |
|--------|---------|
| 2 | ±3° |
| 3 | -4°, 0°, +4° |
| 4 | -6°, -2°, +2°, +6° |

需要变身状态，消耗舰装核心弹药库存，有冷却时间和耐久。

#### Phase 9: 舰装栏 GUI 完善

四项增强：

| 功能 | 实现方式 |
|------|---------|
| 强化栏（放装甲） | ShipCoreMenu 新增 Slot，注册 3 种附加装甲物品 |
| 变身锁定 | Slot.mayPlace/mayPickup 根据变身状态返回 false |
| 负重→移速 | `speedMult = 1.0 - (load/max) * 0.6`，通过 AttributeModifier |
| 装填进度 HUD | RenderGuiLayerEvent.Post，在快捷栏上方绘制进度条 |

#### Phase 10: 数值平衡验证

不写新代码，手动验证三个典型配装场景：

- 小型核心 + 2 小炮 + 双联鱼雷 → ~50% 负重，中等移速
- 大型核心 + 2 大炮 + 四联鱼雷 + 大装甲 → ~89% 负重，极慢但高火力
- 中型核心尝试满载 → 验证超重拒绝机制

DPS 平衡目标：鱼雷纸面 DPS 高于火炮，但受弹药有限、需水面使用、需贴近三重限制。

### 4.3 v0.2.0 中明确排除的内容

声导/磁性鱼雷（追踪 AI）、潜射鱼雷/潜艇核心、鱼雷再装填功能方块、航空/导弹/反潜系统、皮肤/模型渲染、NPC、副本、食物。
