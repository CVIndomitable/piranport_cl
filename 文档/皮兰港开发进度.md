> 平台：NeoForge 21.11.x / Minecraft 1.21.11 / Java 21

## 一、项目概况

### 1.1 策划案来源

原始策划文档《总策划案-20220816-更新港区方块.docx》由"皮兰港"团队编写于 2022 年 7 月，主编 de Nestu，主催轻航独角受。策划基于 Minecraft 1.16.5 + Forge，原有工程代码已丢失。

策划案覆盖子系统：

| 系统 | 策划完成度 | 复杂度 |
|------|-----------|--------|
| 舰装系统（核心、皮肤、负重、变身） | 较高 | 极高 |
| 战斗系统（炮击/雷击/航空/反潜/导弹） | 中等 | 极高 |
| 资源系统（矿物、植物） | 较高 | 低 |
| 方块系统（装饰、功能机器） | 框架 | 中等 |
| 食物-Buff 系统（烹饪、Buff） | 较高 | 中等 |
| NPC 系统（深海、舰娘） | 几乎空白 | 高 |
| 海图系统（副本、深海世界） | 框架 | 极高 |

### 1.2 可行性评估

**整体评价：设计雄心大，落地风险高。**

**主要风险：**
1. **范围过大** — 七八个大系统，每个有大量子模块，相当于中型商业游戏的内容量，但团队是同人社团编制。缺少"最小可玩版本"定义。
2. **技术难度被低估** — 侦察机视角切换（留假人模型+强加载区块）、鱼雷/导弹/战斗机 AI、舰装栏 GUI 深度注入原版背包、三层外观渲染系统（皮肤+船体+模块化武器），每一个都是独立的高难度课题。
3. **大量"待补充"** — NPC 系统、海图副本、深海世界维度、剧情/美术/音乐章节基本为空，策划实际完成度约 50-60%。
4. **数值体系未验证** — 负重上限 112/64/40 的设计缺乏典型配装验证案例。

**亮点：**
- 舰装栏和变身系统逻辑完整，锁定/解锁和弹药库惩罚机制考虑周到
- 烹饪系统（锅的热源判定、产物弹出优先级、碗架渲染）细节充分
- 资源系统克制，没有盲目堆砌新矿物
- 版本分阶段推进思路正确

### 1.3 技术选型

经调研 2026 年 Minecraft 模组开发生态，最终选型：

| 项目 | 选择 | 理由 |
|------|------|------|
| 平台 | NeoForge 21.11.x | 2023 年 Forge 核心团队分裂后的事实标准，16000+ 模组，性能提升显著 |
| MC 版本 | 1.21.11 | 截至决策时最新稳定版，API 成熟，已有 MDK 模板 |
| Java | 21 | NeoForge 要求 |
| 构建工具 | ModDevGradle | NeoForge 标准 |
| 数据存储 | DataComponents | 1.21.x 标准做法，取代旧版 NBT |
| 世界生成 | Data-Driven JSON + BiomeModifier | 无需写 Java 注册 Feature |
| 网络同步 | CustomPacketPayload | NeoForge 21.x 新 API |

## 三、v0.1.0 MVP（已完成）

### 3.1 设计思路

从庞大策划案中裁剪出一条完整的游戏循环：**挖矿 → 冶炼 → 合成装备 → 装进舰装核心 → 变身 → 开炮打怪**

### 3.2 Phase 拆分

| Phase | 内容 | 验证标准 |
|-------|------|---------|
| 1 | 物品/方块注册、Creative Tab | Creative Tab 可见所有物品 |
| 2 | 矾土矿 + 盐块世界生成 | 新世界能找到矿物 |
| 3 | 合成表 + 熔炉配方 | 工作台/熔炉合成正常 |
| 4 | 舰装核心 GUI（DataComponents + Container） | 右键打开 GUI，放入武器/弹药，显示负重 |
| 5 | 火炮投射物 + 变身状态 | 变身后发射炮弹，HE 爆炸 / AP 穿甲 |

### 3.3 关键技术决策

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 物品数据存储 | DataComponents | 1.21.x 标准做法，Codec 序列化更规范 |
| 世界生成 | Data-Driven JSON + BiomeModifier | 不需要写 Java 注册 Feature |
| 网络同步 | CustomPacketPayload + RegisterPayloadHandlersEvent | NeoForge 21.x 新 API |
| 变身系统 | 简化为 Shift+右键切换 | 原策划的动画/无敌帧/buff清除留后续 |

### 3.4 排除内容

鱼雷、航空、导弹、反潜、皮肤/模型渲染、NPC、海图副本、深海世界、食物/Buff、装饰方块、功能方块、树木作物、剧情系统。

## 六、v0.4.0 Aviation（当前开发）

### 6.1 决策背景

v0.3.0 Kitchen 完成后进入 v0.4.0 规划。原路线图中 v0.4.0 为 Deco（装饰方块），经讨论决定将航空系统提前，与 Patchouli 教程手册一起做，Deco 后移至 v0.5.0。

航空系统存在大量历史讨论稿之间的冲突（总策划案、2022.2-4 月多轮讨论纪要、心宿二方案、受受方案等），下面记录逐项裁决结果。

### 6.2 冲突裁决汇总

| # | 冲突点 | 最终决策 | 采纳来源 | 被否决方案 |
|---|--------|---------|----------|-----------|
| 1 | 飞机本质定位 | 飞机 = 武器栏物品，手持右键放飞 | 3.1 结论 + 总策划案 | 心宿二（飞机=弹药，甲板=武器）、受受（掉落物回收） |
| 2 | 整备方式 | 燃料自动补充 + 弹药由编组 GUI 指定 | 混合方案 | 总策划案（GUI 拖拽整备）、3.1 结论（弹药库顺序） |
| 3 | 侦察机 | v0.4.0 不做，只做战斗机和攻击机 | 新决策 | 总策划案（视角切换）、4.4 修订（简化版视角切换） |
| 4 | 火控按键 | 多目标：P 锁定 / O 加选 / I 取消 | 讨论稿结论 | 总策划案（单目标） |
| 5 | 自动索敌范围 | 半径 32 格统一 | 总策划案 | 讨论稿（自身 16 格 + 控制距离 80 格） |
| 6 | 编队 vs 单机 | 编队（单实体多贴图，伤害调整） | 2.12 讨论 | 受受/总策划案（纯单机） |
| 7 | 攻击模式分化 | 四种全做：近战/俯冲/鱼雷/水平轰炸 | 总策划案 | 统一投弹、两种分化 |
| 8 | 弹药选择机制 | 编组 GUI 完全取代弹药库顺序（航空线） | 新决策 | 3.1 结论（弹药库顺序） |

### 6.3 核心游戏循环

```
武器栏装飞机 → 打开编组GUI分组/指定弹种/攻击模式
→ 变身（自动补燃料、挂易燃易爆debuff）
→ 火控锁定目标（P/O/I）
→ 右键放飞编组（直接右键=顺序、数字键+右键=指定）
→ 飞机AI：索敌→攻击→返航
→ 返航后飞机回武器栏、编组GUI中恢复可编辑
```

### 6.4 系统设计

#### 6.4.1 飞机物品

飞机是武器栏物品，与火炮、鱼雷发射器并列，计算负重。

**DataComponent `AircraftInfo`：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `aircraftType` | enum | FIGHTER / DIVE_BOMBER / TORPEDO_BOMBER / LEVEL_BOMBER |
| `fuelCapacity` | int | 燃料容量（决定滞空时间上限） |
| `ammoCapacity` | int | 弹药容量（如天山=1，A2D 天鲨=4） |
| `currentFuel` | int | 当前燃料值（0=未整备） |
| `panelDamage` | float | 面板伤害 |
| `panelSpeed` | float | 面板航速 |
| `weight` | int | 负重 |

**DataComponent `AircraftAmmoInfo`：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `assignedAmmoType` | ResourceLocation | 编组 GUI 指定的弹种 |
| `groupIndex` | int | 所属编组（0-3），-1=未编组 |

#### 6.4.2 编组 GUI

从舰装栏 GUI 中的"编组"按钮打开，变身前后均可编辑。

**固定 4 组**（正航可用 4 组，轻航可用 3 组，由舰装核心决定）。

| 属性 | 说明 |
|------|------|
| 成员 | 从武器栏中的飞机拖入/拖出（引用，不移动物品） |
| 弹药指定 | 为本组指定弹种，放飞时从弹药库扣对应弹药，**未指定则不可放飞** |
| 出击顺序 | 组序号即为默认顺序（1→2→3→4），可调整 |
| 攻击模式 | 集火（全组打同一个火控目标）/ 分散（各自打最近目标） |

飞行中的飞机在编组 GUI 中变灰不可编辑，返航后恢复。

编组数据存储在舰装核心的 DataComponent `FlightGroupData` 中，核心统一管理。

#### 6.4.3 燃料系统

| 环节 | 行为 |
|------|------|
| 变身时 | 自动从弹药库扣航空燃料，填满所有飞机的 `currentFuel`；弹药库无燃料则该飞机不补 |
| 易燃易爆 | 变身状态 + 武器栏有 `currentFuel > 0` 的飞机 → 挂 `FlammableEffect`（受伤加成 + 概率额外爆炸伤害） |
| 放飞检查 | `currentFuel == 0` → 拒绝放飞，HUD 提示"燃料不足" |
| 返航后 | `currentFuel` 清零 |
| 解除变身 | 不清空燃料（下次变身不重复消耗），但移除易燃易爆 debuff |

#### 6.4.4 放飞操作

| 操作 | 行为 |
|------|------|
| 右键 | 按出击顺序放飞下一个未出击的编组 |
| 数字键(1-4) + 右键 | 放飞指定编组 |

放飞时从弹药库扣除编组指定的弹药，弹药不足则提示并拒绝放飞。飞机实体在玩家前方 0.5 格生成。

#### 6.4.5 火控系统

| 按键 | 功能 |
|------|------|
| P | 准星指向生物实体时按下 → 锁定目标；已有锁定时再按 → 切换到新目标 |
| O | 加选目标（追加到锁定列表） |
| I | 取消所有锁定 |

火控数据（`lockedTargets: List<UUID>`）服务端维护，通过 CustomPacketPayload 同步。

与编组攻击模式的交互：集火（FOCUS）全组打锁定列表第一个目标；分散（SPREAD）各自打距自身最近的锁定目标，无锁定时自动索敌 32 格内敌对生物。

#### 6.4.6 AircraftEntity

**状态机：**

```
LAUNCHING（生成后爬升，约1秒）
  → CRUISING（飞向目标/索敌/盘旋）
  → ATTACKING（根据类型执行攻击）
  → RETURNING（飞向玩家）
  → REMOVED（消失，物品回武器栏）
```

**四种攻击行为：**

| 类型 | AI 行为 | 伤害方式 |
|------|---------|---------|
| 战斗机 FIGHTER | 接近目标后近战咬尾，每秒面板伤害 | 直接近战伤害 |
| 俯冲轰炸机 DIVE_BOMBER | 爬升后俯冲接触 | 面板伤害 + 50%概率着火 30s，弹药归零后返航 |
| 鱼雷攻击机 TORPEDO_BOMBER | 贴水面飞行，距 10-16 格投鱼雷 | 复用 TorpedoEntity |
| 水平轰炸机 LEVEL_BOMBER | 爬升至目标上方 32 格，水平投弹 | 新 AerialBombEntity，自由落体+接触 HE 爆炸 |

**编队表现：** 单实体，模型和名称显示为"飞行编队"。伤害 = 面板伤害 × 编队系数（待数值验证）。

#### 6.4.7 防御机制

| 机制 | 触发条件 | 行为 |
|------|---------|------|
| 滞空时间硬上限 | 10 分钟（12000 tick） | 强制 RETURNING / REMOVED + 回武器栏 |
| 距离上限 | 与玩家距离 > 48 格 | TP 到玩家附近，切为 RETURNING |
| 卡死检测 | 连续 60 tick 位置变化 < 0.1 格 | 强制 REMOVED + 回武器栏 |
| 空手右键回收 | 玩家空手右键自己的飞机 | 立即 REMOVED + 回武器栏 |
| 玩家死亡 | PlayerDeathEvent | 所有飞机 REMOVED + 回武器栏 |
| 玩家下线 | PlayerLoggedOutEvent | 同上 |

#### 6.4.8 浮动靶子（测试工具）

`FloatingTargetEntity`：复用原版盔甲架模型，可穿原版盔甲获得护甲值，放置在水面上漂浮。手持靶子物品右键水面放置。用于验证四种攻击模式的伤害数值和护甲减伤。
